\- **Port Kaisola from Electron to Swift-native.** Goals: native feel + kill Electron memory (measured ~880 MB/window today), and share one Swift core across Mac + iPhone. [Full design spec](superpowers/specs/2026-07-20-kaisola-swift-native-design.md) (rev3, two rounds of repo-grounded plan-review). — **strategy approved; awaiting go-ahead on the Phase 0/1 detailed spec.**

**Strategy — hybrid strangler → fully native.** Never a big-bang rewrite. The Electron app stays the daily driver and keeps shipping until the native app is demonstrably better. A native macOS shell (AppKit/SwiftUI on a shared `KaisolaCore` package) grows by *strangling* — it drives the existing battle-tested backends first, then ports each to Swift one at a time. Editor/docs live in a WKWebView (reusing today's React) at first, native later. The one truly durable, socket-speaking backend today is the detached `session-broker` (owns the PTYs); the native app reuses it unchanged first, then replaces it with a Swift PTY daemon last.

**Headline invariant — durable terminal/agent CLI runs survive app update/restart.** An in-flight `claude`/`codex` run keeps running while you update Kaisola; the relaunched app reattaches with continuous scrollback. Bounded honestly: survives app quit/relaunch/update, but *not* broker crash / logout / reboot, and history is retained up to the ~16 MiB spool with an explicit truncation marker.

**Migration phases (each ships; Electron stays the daily driver until parity):** 0) extract `KaisolaCore`, re-base the iPhone app on it · 1) native SwiftTerm terminal spike (observe-only, no ownership theft) · 2) control-plane host + native ACP / windows · 3) editor/docs via WKWebView bridge · 4) daily-driver parity (auth, settings, provider, MCP, companion, updates) · 5) full-native ports + drop Electron.

**Open questions I need your calls on (each blocks its phase):** - **A — control-plane host:** ACP/MCP/companion aren't detached today (they live in Electron main). To reuse them from Swift: build a standalone Node control-plane host for ACP+MCP+companion (broker stays separate) *\[leaning this\]*, pull those Swift ports forward, or run Electron-main headless as a throwaway bootstrap? *(blocks Phase 2/4)* - **B — distribution & sandbox:** App Sandbox (needs security-scoped bookmarks + helper design) vs Developer ID / non-sandboxed (the durable PTY daemon becomes a broad same-user exec capability to secure deliberately)? *(blocks Phase 1 packaging + the daemon)* - **C — data coexistence:** while both apps ship, separate stores + one-way import, or one shared store with a documented write-owner? *(blocks Phase 2/3)* - **D — broker→daemon cutover:** live PTYs can't be handed between processes — drain (old sessions stay on Node until empty), a zero-live-session gate, or keep the Node broker indefinitely? *(blocks Phase 5)* - **Go/no-go:** approve moving to a detailed **Phase 0 + Phase 1** spec (via writing-plans) as the next step?
